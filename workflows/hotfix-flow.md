# Hotfix Flow

긴급 버그/장애 대응용 축약 워크플로우. Planner 단계를 건너뛰고 Backend → QA → CICD 최소 경로로 진행.

## 적용 기준
- P0: 서비스 중단, 데이터 손실/유출, 보안 취약점
- P1: 주요 기능 장애, 다수 사용자 영향

## 전체 흐름
```
[장애/버그 신고 + Stack Trace]
    ↓
[Orchestrator 판단: Hotfix 여부]
    ↓
┌───────────────────────────────────────┐
│ Phase 1: 원인 분석 + 수정 (Backend)   │
│  Input:  BUG-REPORT, Stack Trace      │
│  Output: Fix 코드, 재현 테스트        │
│  QG:     재현 테스트 통과             │
└───────────────────────────────────────┘
    ↓
┌───────────────────────────────────────┐
│ Phase 2: 긴급 검증 (QA)               │
│  Input:  Fix 코드                     │
│  Output: 회귀 테스트 결과             │
│  QG:     재현 실패 + 회귀 통과        │
└───────────────────────────────────────┘
    ↓
┌───────────────────────────────────────┐
│ Phase 3: 긴급 배포 (CICD)             │
│  Input:  Fix 산출물                   │
│  Output: Hotfix PR, Postmortem       │
│  QG:     롤백 준비, 모니터링 강화     │
└───────────────────────────────────────┘
    ↓
[사후 분석 (Postmortem)]
```

## Phase 1: 원인 분석 + 수정 (Backend)
### 입력 자료
- BUG-REPORT 또는 장애 신고
- **전체 Stack Trace 필수** (글로벌 CLAUDE.md 규칙)
- 관련 로그, 메트릭 스냅샷
- 재현 조건

### 절차
1. BUG-REPORT 작성 (아직 없다면)
2. 로컬 재현 → 실패 테스트 코드 작성 (`@Disabled` 제거)
3. 원인 분석 → 최소 침습 수정
4. 재현 테스트 통과 확인
5. 영향 범위 재확인 (동일 결함이 다른 곳에도 있는지)

### 원칙
- **범위 확장 금지**: 핫픽스는 해당 버그만 수정. 리팩토링/개선 별도 PR
- **테스트 먼저**: 재현 테스트 없는 수정 금지
- **근본 원인 문서화**: DECISIONS.md 또는 Postmortem에 기록

### Handoff 조건 (→ QA)
- [ ] 재현 테스트 실패 → 수정 → 통과 확인
- [ ] 영향 범위 명시
- [ ] BUG-REPORT의 Hypothesis 업데이트

---

## Phase 2: 긴급 검증 (QA)
### 절차
1. 재현 테스트 재실행 (실패 → 통과 전환 확인)
2. 회귀 테스트 (해당 기능 + 인접 기능)
3. 보안 관련 결함이면 보안 테스트 추가
4. 부하 관련 결함이면 재현 부하 테스트

### 축소 대상 (생략 가능)
- 신규 TEST-PLAN 작성 (기존 TEST-PLAN에 TC 추가만)
- 전체 부하 테스트 (핫픽스가 성능 관련 아닐 시)

### Handoff 조건 (→ CICD)
- [ ] 재현 테스트 통과
- [ ] 회귀 테스트 전체 통과
- [ ] 신규 결함 없음

---

## Phase 3: 긴급 배포 (CICD)
### 절차
1. `hotfix/<issue-name>` 브랜치 확인
2. 축약 PR 생성 (최소 정보 + 긴급 표시)
3. Reviewer 긴급 호출 (1인 승인 가능 정책 시)
4. CI 통과 후 바로 배포

### 배포 전략
- **소규모 변경**: 즉시 100% 배포 + 강화 모니터링
- **범위 큰 변경**: Canary 10% → 30분 관찰 → 100%
- **DB 마이그레이션 동반**: Forward-compatible 확인 후, 점진 배포

### 롤백 준비 강화
- 직전 버전 이미지 태그 확보
- DB 마이그레이션 롤백 스크립트 미리 실행 가능 상태
- 롤백 명령 사전 작성

### Handoff 조건 (→ Orchestrator)
- [ ] Hotfix PR merge 완료
- [ ] 배포 완료 + 메트릭 정상 확인
- [ ] 강화 모니터링 1시간 이상 관찰
- [ ] Postmortem 문서 초안 작성

---

## Postmortem (사후 분석)
핫픽스 배포 후 **24시간 내** 작성 (Backend + CICD 공동).

### 구성
1. **장애 요약**: 발생 시각, 지속 시간, 영향 범위
2. **타임라인**: 최초 감지 → 원인 파악 → 수정 → 배포
3. **근본 원인**: 5 Whys
4. **재발 방지책**
   - 코드 레벨: 테스트 추가, 아키텍처 개선
   - 프로세스 레벨: 리뷰, CI 게이트 강화
   - 운영 레벨: 모니터링, 알람 튜닝
5. **액션 아이템**: 담당자, 기한 명시

## 작업 로그 (필수)
```markdown
## YYYY-MM-DD [HOTFIX]

### <issue-name>
- [HH:mm] 장애 감지 / 신고
- [HH:mm] Backend 수정 시작
- [HH:mm] QA 검증 완료
- [HH:mm] 배포 완료
- [HH:mm] 메트릭 복구 확인
- Postmortem: docs/postmortems/<date>-<name>.md
```
